垂直 Harness vs 通用 Harness
自建该走什么路线？

针对四个痛点——部署成本高、养护成本高、销售运营开箱即用＋长尾自建 skill、灵活性太高/不稳定——把 X 上的工程共识与你的「Agent 不可能三角」研究拼成一条可落地的自建路线。

2026-05-30bird · 24 检索式下钻 4 串+回复承接 不可能三角篇
先给结论：走「垂直优先 + 半确定性内核 + 长尾 skill 外挂」路线——运行时/编排租，领域资产（memory/state + 分类器 + skill 库）自建。主场景固化做到开箱即用，长尾下放给业务用 skill 自建，全程靠闭环真值 + 失败三分 + 分阶段重试把养护与不稳定压下去。

关键信号 · 先看这 5 条

harness 拆法已清晰：Agent = 模型权重 + harness；harness = 工具＋提示＋重试逻辑＋检索流程。改 harness 不动权重，是你能做的那一半。
「窄域＋单职责」是最被信任的可靠性配方——因为能形成闭环、可对真值，把无声失败变可测。通用 agent 的死穴恰恰是失败不可验证。
自建 vs 租的分界在「资产」不在「编排」：loop/tool-call/编排可租；memory/state、领域分类器、skill 必须自建——这是护城河，也决定你能否离开框架。
部署成本有「调用量阈值」：自托管把账从「按分钟」变「按 GPU 小时」，隐性成本是常驻＋并发；过了地板线才划算，否则托管更便宜。
灵活性别拉满，走「半确定性」：流程约束到可测试，又留口子让模型处理脏输入——稳定性与灵活性的最优折中。

一 · 垂直与通用 harness 差在哪

「每个 Agent 由两大组件构成——模型权重与 harness。改 harness 流派让元-agent 重写任务专用 agent 的脚手架（工具、提示、重试逻辑、搜索流程），权重不动。」 "Every Agent is composed of two main components - Model Weights & Harness… rewrites the scaffold (tools, prompts, retry logic, search procedure) while weights are held fixed." — @kunalbhatia91
维度通用 harness垂直 harness
覆盖面啥都能接只干一类活（你的销售运营主场景）
可靠性低 · 失败不可枚举、无声失败多高 · 可枚举、可对真值
失败可测难（没标准答案）易（闭环、有真值）
提示成本高低（开箱即用）
养护成本高低（可写测试）
灵活性拉满但脆适中、稳定

这正是你前序「Agent 不可能三角」（robustness × generality × cost，arXiv 2605.10907）的落地版：通用性、可靠性、低提示成本三者只能取其二。你要「开箱即用＋可靠」，就必须放弃通用性——走垂直不是妥协，是三角里唯一自洽的一角。

「窄域＋单职责，是我现在唯一信任的 agent 论。冷外呼→约会也是干净闭环，容易对真值。我们跑『暂时性/确定性/永久性』失败三分 + 每阶段重试上限。」 "vertical narrow + one job is the only agent thesis i trust rn… we run a transient/deterministic/permanent split + per-phase retry caps." — @harjjotsinghh

两个可直接抄的抓手：① 闭环可对真值（每场景能自动判「做对没」）；② 失败三分＋分阶段重试上限（暂时→重试 / 确定→换策略 / 永久→转人工，每阶段封顶）——这就是把灵活性约束成稳定性的具体手段。

反方

有人主张会自我改进的通用 agent（SIA）能跑赢专用 agent，但需「改权重」能力且未生产验证，不在你「只改 harness」射程内。@kunalbhatia91

二 · 自建 vs 租 — 分界在「资产」

「框架做 loop、工具调用、编排挺好。我自己攥在手里的是 memory 和 state——这部分会累积，决定你到底能不能离开框架。」 "frameworks are fine for the loop, tool calls, orchestration. the piece i keep in my own hands is memory and state… decides whether you can ever leave the framework." — @val__greg
「你的领域专长 + 我们的 agent 基建（模型、harness、上下文、治理）。工程精力转移到你真正有差异化的地方。」Zscaler SRE：「我建了分类器、错误码映射、探针逻辑，Resolve 负责跑调查。」 — @resolveai

对你是模板级启示：

  • 租掉：模型、运行时、编排循环、基础治理/权限。
  • 自建（护城河）：销售运营的分类器、流程映射、探针/校验逻辑、memory/state、skill 库。
  • 你在「原有智能客服平台升级」——平台就是 harness 底座，别重写底座，要在上面长出领域资产。
两种声音 + 一个代价提醒

偏租：「几乎总是该找框架而非自己写代码。」@OneSeaElephant
偏自建：框架的品味/迭代速度和你不一致时会很难受。@ryanlrlr
代价：闭源模型「用同一套 production harness 做 RL，模型为那套 harness 定制」——自建拿不到这层协同红利，得靠工程补。@scottstts

三 · 逐个击破你的四个痛点

① 部署/托管成本高

「自托管把问题从『按分钟』变成『按 GPU 小时』。隐性成本是保持模型常驻 + 24/7 并发。过了某个调用量地板线自托管才划算；之下，~$0.10/分钟托管反而更便宜。」 — @wellheardai
  • 别一上来自托管模型：先托管 API 算清「调用量×单价」，稳定高过盈亏点 + 有延迟/合规硬约束时再迁。
  • 用 Value = return / cost 看账，警惕 token 浪费（「开满产线却扔掉一半成品」）。@chadwahl
  • 垂直天然省钱：上下文短、工具少、可缓存、可小模型兜底。

② 养护（babysitting）成本高

根源是面太广 + 失败不可测，垂直化本身就是最大减负。

  • 把人移出回路靠「可自动校验」：有闭环真值就能写回归/离线评测，而非人盯。
  • 失败三分 + 分阶段重试上限：自处理暂时性错误、自动转人工处理永久性错误。@harjjotsinghh
  • 好的 auto 模式让你「不用再手把手牵它过 API 认证迷宫」。@YotamBlu
  • 配端到端 tracing：养护从「人肉巡检」变「看 dashboard」。

③ 销售运营开箱即用 + 长尾自建 skill

双层架构：主场景固化（开箱即用）+ 长尾 skill 外挂（业务自建）。

「先规划、要你确认、再开干，横跨 Workspace/Slack/GitHub 和任意自定义连接器；可复用 skill，每步可见。」 — @mistralvibe（Mistral Vibe 工作模式）
「我现在工作里每件事都写成 agent skill——高度具体、单一目的，每周省好几个钟头。」例：博客→官方社媒风格，为不同平台写不同帖、知道怎么 @ 公司、再上传到排期工具。 — @ctnicholasdev

给销售运营提供「skill 模板 + 录制/捏制入口」，把长尾流程沉淀成窄、专、可复用的 skill。进阶：agent 甚至能自己造 skill（大重构里自动给子 agent 写工具）。@poteto

⚠️ 治理：开放业务自建 = 灵活性下放给非技术用户，必须配沙箱、权限范围、审核，否则脏 skill 污染主流程。

④ harness 灵活性太高 / 不稳定

「半确定性 harness 是个好方向：工作流约束到足够可测试，又留出灵活性让模型应对脏输入。」 "Semi-deterministic harnesses… constrained enough to test, but flexible enough for the model to handle messy inputs." — @Gsdata5566
  • 骨架确定、关节柔性：主流程写成确定性状态机（可测可回归），只在「理解脏输入/生成措辞」处放模型自由度。
  • 护栏 + 分阶段重试上限：超限即降级/转人工，杜绝无限打转。
  • 闭环校验兜底：每步有真值检查，跑偏立即可见。

四 · 给你的自建路线（分期）

一句话：垂直优先做主场景的半确定性 harness，租运行时/编排、自建领域资产，主场景开箱即用、长尾下放 skill，靠闭环真值＋失败三分＋分阶段重试压住养护与不稳定。

第一期
稳住主场景：选 1–2 个高频闭环（线索跟进→约访 / 工单分类→回填），确定性骨架 + 模型关节 + 真值校验 + 回归测试。先托管 API，不自托管。
第二期
开箱即用 + 可观测：plan→确认→执行的交互壳 + 步骤可见 + tracing dashboard，业务零提示跑通主流程，养护转看板。
第三期
长尾 skill 平台：给业务 skill 模板 + 捏制入口 + 沙箱/权限/审核，把长尾沉淀成可复用 skill；skill 库 + 领域分类器即护城河。
持续
盯调用量是否过自托管盈亏线；通用化呼声出现时回「不可能三角」自检——是否在拿可靠性/低提示成本换不该要的通用性。
