Codex/ˈkoʊdeks/OpenAI 的编程智能体产品线，现含 App / CLI / SDK；与 2021 年同名旧代码模型不是一回事。
套壳中文俗语：拿现成开源/产品改个皮（换名字、图标、界面）当自己的。本文分两层——① fork 改皮 ② 接别的模型做包装层。
CLICommand-Line Interface，命令行界面工具。
SDKSoftware Development Kit，软件开发工具包，供开发者二次集成。
fork/fɔːrk/把一个开源仓库完整复制到自己名下，在此基础上改。
Apache 2.0宽松型（permissive）开源许可证，允许商用、修改、闭源再分发，只需保留版权与许可声明。
AGPL-3.0「传染型」（copyleft）许可证，连网络服务也须开源你的修改，常用来「防白嫖」。
permissive / copyleft宽松型 vs 传染型，开源许可证的两大阵营。
agentic/eɪˈdʒentɪk/代理式的；AI 能自主规划、调用工具、多步执行，而非被动一问一答。
harness/ˈhɑːrnɪs/把模型「套」起来干活的外层框架：负责工具调用、上下文管理、循环执行。Codex CLI 本质就是一套 harness。
Responses APIOpenAI 较新的接口规范，支持有状态对话与工具调用；Codex 默认依赖它。
Chat Completions更老更通用的 OpenAI 风格接口；绝大多数第三方/开源模型只兼容这一种。
tool-calling工具调用，模型按约定格式请求执行外部函数/命令的能力。
computer use / browser use让智能体操作电脑桌面 / 浏览器的高级能力。
base_url配置里指向「模型从哪个服务地址取」的字段，换供应商的关键开关。
proxy/ˈprɒksi/代理层；这里指夹在 Codex 与第三方模型之间、把 Responses 接口翻译成 Chat Completions 的中间件。
model-agnostic模型无关（不绑定特定模型，可任意替换）。

争议与开放问题

开放问题「套壳」的法律边界 vs 品牌边界

Apache 2.0 在代码上给足自由（可商用、可闭源分发），但「Codex」是商标，商标≠版权——你能用它的代码，不能直接打它的牌子。X 上暂无围绕官方 Codex 商标维权的讨论，属开放问题（推断需另查 OpenAI 品牌/使用条款）。
接第三方模型后的能力完整性：computer use / browser use 受限是当下事实，OpenAI 是否会逐步对自定义 provider 开放，未见官方承诺，待观察。
「学习版套壳」与「商业套壳」的差距：现有公开案例都强调「非商业、学习用」。真把套壳产品商业化卖钱的合规与口碑风险，X 样本里没人现身说法，属空白区。
新旧 Codex 混淆风险：网上仍有大量讲 2021 旧 Codex 的旧文，检索「Codex 开源」时务必确认对象是新工具链。
