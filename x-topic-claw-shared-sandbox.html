让一堆 claw 共享底座
agent 沙箱的密度与隔离

把部署模式从「一个 claw 独占一个容器」改成「一堆 claw 通过分布式文件夹 + 沙箱共享底座」降本——方向对吗、业界怎么做、怎么选型、坑在哪。这是「自建 harness」痛点①（部署密度，非自托管模型）的专项展开。

2026-05-30bird · 12 检索式下钻 3 串承接 harness 篇
先给结论：方向对，但别简化成「全部共享」。正确姿势是 按威胁模型分层隔离（tier）+ 用 COW/快照/池化把闲置成本压没，而不是「一 claw 一容器 24h 常驻」。你的销售运营是 第一方、跑自己的 skill（非陌生人代码），可用 高密度的轻隔离（共享内核 + gVisor），把 microVM 留给真要跑不可信代码的场景。

关键信号 · 先看这 5 条

共享内核容器从来不是安全边界——「只是我们集体假装是墙的便利边界。agent 栈在用十年前 Lambda 就解决过的方式重新发现多租户隔离。」降密度前先想清威胁模型。
microVM ≠ 万能沙箱：「单靠它的隔离不够，把 agent 关进 microVM 也可能越狱。」隔离是系统工程，不是套层 VM 就完事。
分层隔离是业界范式：Anthropic 三产品三隔离（gVisor / OS 沙箱+人审 / 完整 VM），原则「先做环境隔离、再谈模型层引导」「隔离强度匹配用户能力」。
密度可以很高且不必上 SaaS：「每个沙箱进程作为独立 gVisor 线程，和现有 k8s pod 同节点跑，自动扩。不需要外部 Sandbox SaaS。」
降本关键是别让闲置实例空转：CRIU+overlayfs 亚 500ms 快照恢复、COW 每任务分叉、生命周期绑定 scope 用完即毁——而非一 claw 一容器常驻。

一 · 为什么「一 claw 一容器」默认但贵

独占容器隔离最省心，但代价是：闲置空转（claw 多数时间在等模型/等人，容器却 24h 占资源）、冷启动浪费（频繁拉起/销毁）、存储重复（N 个 claw = N 份镜像）。

「编码 agent 不该因为弄坏测试环境就等 Docker 重启 3 秒。」——这种等待和重复就是密度低的直接成本。 — @JainishP8

你的直觉（共享底座降本）是对的，问题只在 怎么共享才既省又安全。

二 · 隔离技术光谱：密度↔隔离是一条曲线

「sandbox 只是『隐含安全的隔离』，没规定实现——可以是容器（gVisor），也可以是 VM（Firecracker）。」 — @yegle
档位技术密度/成本隔离适用
共享内核容器Docker/namespace最高最便宜弱（非安全边界）全第一方可信负载
用户态内核沙箱gVisor/runsc高中-强服务端半可信（claude.ai）
微型虚拟机Firecracker/Kata中（仍轻量）强（Lambda 级）多租户/不可信代码
OS 沙箱Seatbelt/bubblewrap看宿主中（+人审）跑在用户机（Claude Code）

两条刻进脑子的警告：

  • 共享内核容器便宜但不是安全墙——claw 互不信任或跑外部代码时别只靠它。@0xToxSec
  • 即便 microVM/Kata 也有逃逸 CVE，要配 egress 过滤 + 凭证不进 VM + 策略 chokepoint 才完整。@iximiuz · @Techstrongai

三 · 共享底座的三件套（降本的实处）

3.1 分布式/虚拟文件系统（你说的「分布式文件夹」）：基础镜像只读共享一份，每 claw 只持增量写层（overlayfs/COW），省存储且「丢弃一个 claw」近乎免费。
「写时复制让 claude-code 工作流真正安全：每任务一分支，测破坏性迁移，出岔子就扔掉。」业界已有 agent 专用虚拟 FS：chay2u 用 Turso 的 agentfs（受 Cloudflare vibesdk 启发）。 — @trevorlasn · @chay2u
3.2 快照 / fork / 预热池（杀掉空转与冷启动）：不让闲置 claw 常驻，冻结成快照、秒级恢复，或预开基础实例 fork。
「用 CRIU + overlayfs 把 Checkpoint/Restore 做到亚 500ms。」赢家级 harness 的标准是「持久化会话、干净 fork、重启不清空上下文」。 — @JainishP8 · @m13v_
3.3 生命周期绑定（杜绝僵尸实例漏成本）：
「沙箱生命周期绑定到 scope，scope 关闭即销毁——没有悬挂实例、没有泄漏。」 — @janschultecom

三件套合起来 = 共享只读基础层（省存储）+ 快照/池化（省闲置与冷启动）+ 用完即毁（不漏成本）。这才是「一堆 claw 共享底座、成本低」的工程实现。

四 · 选型：按威胁模型分 Tier（别一刀切）

Anthropic 三产品三隔离 +「先做环境隔离、模型层引导次之」「隔离强度匹配用户技术能力」。Cloudflare 也按用例分不同「执行 Tier」。 — @buzea200 转 Anthropic 工程博客 · @_bgiori

落到你的销售运营 claw：

  • claw 是第一方、跑你平台的 skill（非陌生人任意代码）→ 威胁温和 → 高密度轻隔离：共享内核 + gVisor 进程级，一节点塞很多 claw（参照 @christopherwxyz 的 GKE+gVisor 样板）。
  • 只有开放「业务自建 skill 跑任意代码 / 跑用户上传脚本」这类不可信执行时，对那部分 claw 才升级到 microVM + egress 过滤。
  • 接上篇治理提醒：长尾自建 skill = 引入不可信执行面，单独划到强隔离 tier。

五 · 别过度工程化 + 租 vs 自建

「研究 agent 沙箱架构好几个月、找了一堆做 Firecracker 的团队，结果发现我们大多数人在解决根本不存在的问题。」 — @chay2u
  • 先按真实威胁模型定 tier，再选技术，别上来就 Firecracker 全家桶。
  • 能租就租（呼应上篇「运行时租」）：Vercel Sandbox（Docker-in-sandbox、跨会话持久化）、E2B、Modal、Daytona、Fly Machines、Docker 官方 microVM 沙箱。
  • 自建的理由：若已有 k8s 底座、要的就是密度，自己上 gVisor 层「不需要外部 Sandbox SaaS」性价比高。@christopherwxyz

给你的落地建议

一句话：把「一 claw 一容器常驻」换成「共享只读基础层（overlayfs/agentfs）+ gVisor 轻隔离塞高密度 + CRIU 快照/warm pool 干掉空转 + 用完即毁」；按威胁模型分 tier，温和内场景走高密度轻隔离，跑不可信代码才上 microVM。

  • 定 tier：内场景跑自家 skill → 共享内核+gVisor；跑不可信代码 → Firecracker+egress 过滤。
  • 共享文件层：基础镜像 overlayfs 只读共享，每 claw 只写增量层；评估 Turso agentfs。
  • 干掉空转：CRIU+overlayfs 秒级快照恢复 + warm pool；闲置冻结而非常驻。
  • 生命周期绑 scope：会话结束即销毁，杜绝僵尸；但保留「持久化会话 + 干净 fork」，重启不丢上下文。
  • 隔离配套：凭证不进沙箱、egress 过滤、工具调用走统一鉴权/日志 chokepoint。
  • 先租后建：先用 E2B/Vercel Sandbox/Modal 验证形态与成本；已有 k8s 且追密度再自建 gVisor 层。
  • 别过度工程化：威胁温和就别上 Firecracker 全家桶。
