CLI vs MCP —— 以及为什么两边都错了

两派各执一词吵了整个 2025 年，直到 Anthropic 从第一性原理重构：真问题从来不是协议，而是「一开始就把所有工具灌进上下文」。

⌨️ CLI 派的理由

  • agent 训练时就「认识」grep / cat / git，零学习成本
  • 命令直接映射任务，不绕抽象层
  • MCP 一个 5 服务器配置开工前先烧 55K token

🔌 MCP 派的反击

  • CLI 在多租户（multi-tenant）应用上会崩
  • 没有类型契约，agent 只能猜输出
  • 面对陌生 API，agent 浪费回合解析文本
2025.11 Anthropic 的重构 · Code Mode

不再让模型「通过上下文调工具」，而是让模型写代码、通过运行时（runtime）调工具。工具住在 runtime 里，模型只看到它 import 的部分。Bash（处理已装二进制）+ typed imports（专有 API，类型随 import 才加载）——「MCP 的类型契约 + CLI 的惰性加载，合在一个运行时里」。@akshay_pachaar

📌 别误读成「MCP 已死」：Anthropic 报告 MCP SDK 下载量从年初 1 亿涨到 3 亿，是增长最快的 agent 基础设施。死掉的只是「一次性加载所有工具」这个坏习惯。

🧩 共识：给 agent 的工具要为机器设计

人类 CLI 的输出（彩色、日志、进度条）原样灌进上下文就是噪声 + 烧钱。

Printing Press（CLI 工厂+库）直言：大多数 API、MCP、官方 CLI 给 agent 用都很烂，于是做 agent-native CLI——快、本地、SQLite backed，跨 Claude Code/Codex/OpenClaw/Hermes 通用。

🏗 全景：CLI 只是「harness 工程」里的一块

你觉得像魔法的体验，背后是 harness 工程，不是模型本身。

Claude Code 远不止「一个调用 claude 的 CLI」——是 6 层 harness（输入/知识/执行/集成/多agent/可观测），模型只是循环里一个节点；执行层用带类型的注册表分发 bash/read/write/grep。

@akshay_pachaar · 706♥

别混淆四原语：Skills=知道什么 / MCP=怎么连 / Hooks=何时自动 / Subagents=谁干活。混为一谈会导致生产环境崩掉的架构。

@NainsiDwiv50980 · 385♥

MCP 给手，Skill 给肌肉记忆：MCP 解决「连接问题」却没解决「使用问题」——给 50 个接好的工具，agent 仍可能不知何时用哪个。

@_avichawla · 304♥

harness > 模型：4 个跟模型无关的修复（前缀缓存保温=TTFT 从 6-8s 降到 1s 内、规范化模型 ID、能力位协商、思考模式兼容）让开源模型逼平 Opus。「模型没变聪明，是 harness 不再丢掉它的成果」。

@MrAhmadAwais · 499♥

「通用 agent」不存在：针对某 Task 定制 harness 就能跑赢任何默认组合；最终都收敛到好的提示和工具设计（打包成 Skill）。正进入「子 agent 作为工具」的拆分时代。

把人类 CLI 直接给 agent——给眼睛看的输出灌进上下文=噪声+烧钱。@mvanhorn

一次性加载所有工具描述——单流程可膨胀到 150K token。@akshay_pachaar

终端原始输出直灌上下文——/context 开局就 >20% 说明在漏 token。@DeRonin_

MCP 工具响应不做摘要——单个页面读取就吃数千 token，应 sandbox 到 SQLite 只回摘要。@DeRonin_

harness 偷懒 → 错怪模型——前缀缓存被负载均衡偷走等，让模型「输掉」本该赢的评测。@MrAhmadAwais

让 agent 碰生产凭证 / 无独立备份——「别在 agent 能拿到生产凭证的地方跑它」。@simonw

沙箱甩给用户自己搞——「每个 agent 框架都该开箱自带一流沙箱」，现状是留给用户、且很难做好。@simonw

CLI 自动放行命令留提权口子——Claude Code 修过「裸环境变量赋值被自动批准」的 Bash bypass。@ClaudeCodeLog
